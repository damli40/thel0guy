What bad docs cost a coding agent. 40 runs, measured.

Integrations don't start with a human reading your docs anymore. They start with a developer pointing Claude Code or Cursor at those docs and letting it build. I wanted the dollar cost of that, so I ran the experiment. Bad context cost about 6x per working integration, at 5x the clock.

The setup

One real integration task on a smart-contract toolchain. One model. I gave it four different context setups and ran each one many times, 40 scored runs in total.

  • Bare model: no docs, no tools. Just the model and the task.
  • + retrieval tool: the model can search a corpus when it decides it needs to.
  • + curated docs: a clean, agent-readable doc layer in context.
  • + docs and retrieval: both together.

A run passed or failed by one rule: rerun the toolchain's test suite in a clean container and check the result. What the model claimed about its own work counted for nothing. Every cost and timing number came from the provider's own billing fields, summed per run.

The numbers

Median cost and time per working integration, so a cheap run that failed the test suite earns no credit:

setupmedian cost / working integrationmedian time
Bare model$0.57311s
+ retrieval tool$0.10156s
+ curated docs$0.0960s
+ docs and retrieval$0.0970s

The bare model still shipped working integrations. It just paid about 6x more per one and took 5x as long, grinding through turns to reconstruct what a good doc would have handed it in one read. The token cost is real and it never shows up on anyone's dashboard as a docs problem.

The part that matters for your docs

I ran a second task on purpose outside the docs' coverage. Curated docs alone lost nearly all their edge there, because the answer was not in them and the model had no way to go looking. Docs plus retrieval stayed cheapest and fastest, because the model could reach past the doc when the doc ran out.

That boundary, the line where your docs stop answering and the agent starts guessing, is the expensive part. It is invisible until someone measures it. Finding it in your docs, in dollars, is what the agent cost audit does.

What this is and isn't

  • One toolchain, one model, one point in time. The numbers are directional, not a universal constant.
  • A controlled setup, not your repo. Your docs will have their own gap map, which is the whole point of running the audit on them.
  • Every figure is summed from provider billing fields against a test-suite oracle. Measured, not estimated. I would rather show you the method than a bigger headline.
